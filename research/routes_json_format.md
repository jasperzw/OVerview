# routes.json — Format Documentation

`web_extension/data/routes.json` is generated by `scripts/build_route_geometries.py` from
the OpenOV lijnnetkaart shapefile (`20260511.zip`).

---

## Top-level structure

```json
{
  "NS:TRAIN":   [ ...route objects... ],
  "GVB:TRAM":   [ ...route objects... ],
  "RET:METRO":  [ ...route objects... ],
  ...
}
```

Keys are `"OPERATOR:MODE"` strings. Operator is the normalized OV-chipkaart `pto` code
(e.g. "NS", "GVB", "Arriva"). Mode is one of `TRAIN`, `TRAM`, `METRO`.

Operator normalization applied in `build_route_geometries.py`:

| Shapefile name      | JSON key operator |
|---------------------|-------------------|
| NS International    | NS                |
| R-net NS            | NS                |
| Blauwnet Arriva     | Arriva            |
| RRReis Arriva       | Arriva            |
| Blauwnet Keolis     | Keolis            |
| RRReis Keolis       | Keolis            |
| R-net Qbuzz         | Qbuzz             |

## Route object

```json
{
  "publicCode": "Intercity",
  "name":       "Amsterdam Centraal <-> Den Haag Centraal IC2100",
  "from":       "Amsterdam Centraal",
  "to":         "Den Haag Centraal",
  "coords":     [[52.37900, 4.89980], [52.37300, 4.88750], ...]
}
```

| Field        | Type            | Notes |
|--------------|-----------------|-------|
| `publicCode` | string          | Line/service code. NS trains: "Intercity" or "Sprinter". Urban: line number as string ("12", "E"). |
| `name`       | string          | Human-readable from shapefile NAME field. Format: "A <-> B [service_code]" or "Lijn N". |
| `from`       | string          | Parsed terminal station name (A in "A <-> B"). Empty for urban lines with "Lijn N" names. |
| `to`         | string          | Parsed terminal station name (B in "A <-> B"). Empty for urban lines with "Lijn N" names. |
| `coords`     | [[lat,lon], …]  | WGS-84 coordinates in **[latitude, longitude]** order (Leaflet-ready). Simplified with Douglas-Peucker (ε ≈ 30 m). |

---

## Coordinate system

- **Projection:** WGS-84 (EPSG:4326) — matches Leaflet directly, no reprojection needed.
- **Order:** `[latitude, longitude]` — already swapped from shapefile's `(longitude, latitude)` order.
- **Precision:** 5 decimal places (~1 m).

## Matching algorithm (app.js)

`findRouteGeometry(trip, fromCoord, toCoord)` in `app.js`:

1. **Detect mode:** from `trip.product` text ("intercity" → TRAIN, "tram" → TRAM, etc.) or from `trip.operator` (NS → TRAIN, HTM → TRAM).
2. **Lookup pool:** `ROUTES[operator + ":" + mode]`
3. **Filter by publicCode:** extract from product text ("Intercity", "12", "E"), filter pool to matching `publicCode`. Fallback to full pool if filter yields nothing.
4. **Score each route:** find closest index to `fromCoord` and `toCoord` in the polyline. Reject if either is >5 km away. Score = sum of squared distances.
5. **Pick best:** lowest score wins.
6. **Clip:** slice polyline from closest-to-`from` index to closest-to-`to` index.
7. **Direction:** reverse slice if `from` index > `to` index (route stored in opposite direction).

Fallback: straight crow-flies polyline. Shown with `〜` mark in popup.

## Regenerating

```bash
# Download the latest lijnnetkaart from http://data.openov.nl/lijnnetkaart/
# Place the zip in the project root as e.g. 20260511.zip

pip install pyshp --break-system-packages  # first time only
python scripts/build_route_geometries.py 20260511.zip
```

The script writes to `web_extension/data/routes.json`. Run after each new lijnnetkaart release
(typically every few months when the network changes).
